# 系统架构与技术决策记录

本文档记录项目的技术选型、架构设计和优化方向，作为系统演进的历史参考。

---

## Rerank 技术分析

## 什么是 Rerank？

Rerank（重排序）是一种两阶段检索策略：

```
第一阶段：粗检索（BM25/向量检索）→ 返回 Top-K 候选（如 100 个）
第二阶段：精细重排序（Cross-Encoder）→ 对候选重新打分，返回 Top-N（如 10 个）
```

## 当前项目的检索流程

```
用户查询 → LLM 扩展关键词 → BM25 多关键词检索 → 合并去重 → Token 预算控制 → 输出
```

## 是否适合本项目？

**不太适合**，原因如下：

| 维度 | 分析 |
|------|------|
| 数据规模 | 单个文档 ~150K tokens，BM25 已经足够高效 |
| 精度需求 | 财务报表检索是"精确匹配"场景，关键词命中即可 |
| 延迟要求 | Rerank 需要额外调用模型，增加 100-500ms 延迟 |
| 成本考虑 | Cross-Encoder 调用有 API 成本 |
| 当前效果 | LLM 扩展 + BM25 已能召回三大财务报表 |

## Rerank 适用场景

Rerank 更适合：

1. **大规模语料库**：百万级文档，需要高召回率
2. **语义匹配场景**：问答、对话、推荐系统
3. **向量检索补充**：向量检索精度不足时，用 Rerank 精排

## 本项目更好的优化方向

| 优化方向 | 说明 |
|----------|------|
| 查询扩展优化 | 改进 LLM prompt，生成更精准的关键词 |
| 表格权重调优 | 根据表格类型动态调整分数权重 |
| 多轮检索 | 根据首次检索结果，动态补充查询 |
| 上下文窗口优化 | 智能选择表格的关键行列，而非整表 |

## 结论

对于单文档、关键词匹配为主的财务报表检索场景，Rerank 收益有限，成本增加。当前架构（LLM 扩展 + BM25 + 表格权重）已经能满足需求。

如果未来扩展到多文档、语义检索场景，可以考虑引入 Rerank。

## 摘要分块技术分析

### 什么是摘要分块？

摘要分块是一种语义驱动的文档切分策略：

```
原始文档 → 逐段生成摘要 → 基于摘要相似度合并/切分 → 生成语义完整的块
```

与传统分块的对比：

| 方法 | 分块依据 | 优点 | 缺点 |
|------|----------|------|------|
| 固定长度 | 字符数/Token 数 | 简单高效，块大小可控 | 可能切断语义单元 |
| 段落分块 | 换行符 | 保留段落完整性 | 段落长度差异大，块大小不均 |
| 摘要分块 | 语义相似度 | 语义完整，检索更精准 | 需要调用 LLM，成本高、延迟大 |

### 是否适合本项目？

**不太适合**，原因如下：

| 维度 | 分析 |
|------|------|
| 文档结构 | 财务报告有明确的章节结构（标题、表格），天然适合结构化分块 |
| 表格处理 | 表格是财务报告的核心，摘要分块难以处理表格语义 |
| 成本考虑 | 每个段落调用 LLM 生成摘要，成本高（~150K tokens 文档可能需要数百次调用） |
| 延迟要求 | 预处理阶段需要等待摘要生成，不适合快速迭代 |
| 当前效果 | 表格感知分块 + BM25 已能准确定位表格和章节 |

### 摘要分块适用场景

摘要分块更适合：

1. **长篇叙事文本**：小说、新闻、论文，语义边界模糊
2. **问答对构建**：将文档转为 Q&A 格式，用于 RAG 微调
3. **语义去重**：识别重复或相似内容
4. **多模态文档**：图文混排，需要理解整体语义

### 本项目的分块策略

当前采用"表格感知分块"，已针对财务报告优化：

```
文档 → 识别表格位置 → 表格作为独立块 → 非表格区域按段落分块 → 添加表格标题上下文
```

**优势**：
- 表格完整性：整个表格作为一个块，不会被切断
- 标题关联：自动关联表格前的标题
- 零额外成本：无需调用 LLM

### 未来可能的优化方向

如果需要更精细的语义理解，可以考虑：

| 方向 | 说明 | 成本 |
|------|------|------|
| 表格摘要 | 只对表格生成摘要，用于检索 | 中等（表格数量有限） |
| 章节摘要 | 只对章节标题生成摘要，增强检索 | 低（章节数量少） |
| 混合索引 | BM25 + 向量检索，向量用于语义补充 | 中等（一次 embedding 调用） |

### 推荐路线：Parent-Child / Summary RAG

如果未来 BM25 真的出现检索瓶颈，推荐采用"小块检索，大块喂给模型"的策略：

```
表格完整保留（Parent）→ 生成 50 字简短标签（Child）→ Child 参与检索 → 命中后返回完整 Parent
```

**示例**：
```
Parent（完整表格）：CSV 格式的现金流量表（~2000 tokens）
Child（简短标签）："2025年度合并现金流量表及补充资料"
```

**检索流程**：
1. 用 Child 标签参与 BM25 检索
2. 一旦命中，直接返回完整 Parent 表格
3. 完整表格塞入 64K 上下文

**优势**：
- 检索精准：标签简洁，BM25 匹配更准确
- 内容完整：返回完整表格，无信息丢失
- 成本可控：只需对表格生成标签（数量有限）

### 内网环境设计原则

在银行内网环境下，系统设计应遵循：

> **"越少调用 LLM 做预处理，系统的鲁棒性就越高"**

| 原则 | 说明 |
|------|------|
| 减少预处理依赖 | 预处理阶段调用 LLM 越多，失败点越多 |
| 优先离线处理 | 能离线生成的标签/摘要，不要实时调用 |
| 降级方案 | LLM 不可用时，系统仍能基本运行（BM25 单独工作） |
| 轻量模型优先 | 必须调用时，优先使用轻量模型（如 7B）而非大模型 |

### 结论

对于结构化强、表格为主的财务报告，摘要分块收益有限，成本较高。当前"表格感知分块"策略已能保证语义完整性。

如果未来处理非结构化文本（如管理层讨论、分析师报告），可以考虑引入摘要分块。
